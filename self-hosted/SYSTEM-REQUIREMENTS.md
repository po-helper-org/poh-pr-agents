# Системные требования: отказоустойчивое ядро PR-Agent (СА / системная высота)

> Высота ниже архитектуры: **как** устроено ядро, обрабатывающее события Issue/PR
> без тихих падений. Каждое СТ атомарно, проверяемо и трассируется к контракту
> надёжности из issue #1 и к фазе из [`SCALE-PLAN.md`](SCALE-PLAN.md).
> Оформлено в методологии sa-helper (навык установлен в po-helper; здесь — прямой
> эквивалент того же формата).

## Якоря контракта (из #1)

| Код | Контракт |
|---|---|
| К-1 | Нет тихих падений: 100% событий → наблюдаемое терминальное состояние (`done` \| `dead_letter` + оповещение) |
| К-2 | Гарантированное eventual-завершение: ретрай до успеха или эскалация; ничто не теряется молча |
| К-3 | Ограниченная задержка: p95 «событие → ревью» < 10 мин под целевой нагрузкой |
| К-4 | Идемпотентность: at-least-once доставка + идемпотентный эффект (без дублей ревью) |
| К-5 | Наблюдаемость: строка состояния, метрики и трассировка на каждое событие |

## §0. Границы СТ

**В зоне:** ingress, durable queue, state store, worker pool, LLM gateway, publisher,
dead-letter + оповещение, reconciliation sweeper, observability.
**Вне зоны (не в этих СТ):** сам анализ pr-agent (describe/review/improve — переиспользуется
как есть), выбор инфра-платформы/брокера (решение §5 issue #1), UI/дашборды сверх метрик.

## §1. Компонентная архитектура

Топология и диаграмма — в [`SCALE-PLAN.md` §2](SCALE-PLAN.md). Ниже — требования к
каждому компоненту.

## §2. Ingress (приём webhook)

| СТ | Требование | ← |
|---|---|---|
| СТ-1 | Проверяет HMAC (`X-Hub-Signature-256`); несовпадение → `401`, событие не принимается | К-1 |
| СТ-2 | Извлекает `X-GitHub-Delivery` как idempotency-key; повтор того же id не создаёт второй обработки (dedup через state store), возвращает `200` | К-4 |
| СТ-3 | Пишет событие в очередь + строку `received` в state store **до** возврата `200`; если запись в очередь не удалась → `5xx` (GitHub повторит доставку), не теряет молча | К-1, К-2 |
| СТ-4 | Не делает тяжёлой работы (ни LLM, ни GitHub API); ответ p95 < 500 мс | К-3 |
| СТ-5 | Stateless, горизонтально масштабируется за LB | К-3 |

## §3. Durable queue

| СТ | Требование | ← |
|---|---|---|
| СТ-6 | Durable: переживает рестарт брокера и потребителя; неподтверждённое сообщение возвращается по visibility-timeout | К-2 |
| СТ-7 | Партиции по `(installation_id, repo)` — честность, одно «тяжёлое» репо не голодит остальные | К-3 |
| СТ-8 | Payload содержит: `delivery_id, event_type, repo, number, head_sha, command, attempt_count` | К-4 |
| СТ-9 | После N попыток (конфиг, дефолт 5) сообщение → dead-letter, не удаляется | К-1, К-2 |

## §4. State store и машина состояний

| СТ | Требование | ← |
|---|---|---|
| СТ-10 | Состояния `{received, queued, processing, done, failed, dead_letter}`, переходы только по разрешённому графу | К-1, К-5 |
| СТ-11 | Ключи: idempotency `delivery_id` + бизнес `(repo, number, head_sha, command)` | К-4 |
| СТ-12 | Фиксирует timestamp каждого перехода и `attempt_count` (основа latency-SLO и reconcile) | К-3, К-5 |
| СТ-13 | Событие не остаётся вне терминала (`done`\|`dead_letter`) дольше T (конфиг, дефолт 30 мин) без перевода в `failed`/повторной постановки | К-1, К-2 |

## §5. Worker pool

| СТ | Требование | ← |
|---|---|---|
| СТ-14 | Stateless; тянет сообщение → `processing` → запускает анализ pr-agent с per-task таймаутом (конфиг) | К-3 |
| СТ-15 | Успех → `done` + ack; исключение/таймаут → nack → повторная доставка с backoff; исчерпание попыток → `dead_letter` | К-1, К-2 |
| СТ-16 | Идемпотентен по бизнес-ключу: перед публикацией проверяет, не опубликован ли результат для `(number, head_sha, command)`; если да → no-op `done` | К-4 |
| СТ-17 | Падение воркера в `processing` → сообщение переедет к другому по visibility-timeout | К-2 |
| СТ-18 | Автоскейл по глубине очереди / возрасту сообщений | К-3 |

## §6. LLM Gateway

| СТ | Требование | ← |
|---|---|---|
| СТ-19 | Все LLM-вызовы идут через gateway; воркеры не ходят к провайдеру напрямую | К-3, К-5 |
| СТ-20 | Per-key token-bucket + очередь ожидания; на `429` — backoff, не мгновенный бросок ошибки | К-2, К-3 |
| СТ-21 | Circuit breaker на провайдера: серия ошибок → размыкание, быстрый fail/переключение, метрика `circuit_open` | К-1, К-5 |
| СТ-22 | Failover по списку провайдеров/ключей. **Текущее ограничение:** один ключ Z.AI → failover вырожден в ретрай (issue #1; второй провайдер — предусловие Фазы 4) | К-2 |
| СТ-23 | Кэш идентичных промптов (хеш `prompt+model`, TTL) → меньше стоимость и нагрузка | К-3 |
| СТ-24 | Тиринг: матрица «команда → модель» (describe → лёгкая, review → тяжёлая) | К-3 |

## §7. Publisher (идемпотентный эффект)

| СТ | Требование | ← |
|---|---|---|
| СТ-25 | Публикует upsert-ом: правит существующий бот-коммент по `(number, command)`, а не создаёт новый → нет дублей при ретраях | К-4 |
| СТ-26 | Отдельный идемпотентный шаг с собственным ретраем на транзиентных ошибках GitHub API | К-2 |

## §8. Dead-letter + оповещение

| СТ | Требование | ← |
|---|---|---|
| СТ-27 | Любое событие в dead-letter → (а) видимый коммент в PR/issue «обработка не удалась: <класс ошибки>, эскалировано», (б) метрика `dead_letter_total`, (в) [опц.] внешний алерт (issue #2) | К-1 |
| СТ-28 | Dead-letter не финал: событие доступно для ручного/автоматического повторного запуска | К-2 |

## §9. Reconciliation sweeper

| СТ | Требование | ← |
|---|---|---|
| СТ-29 | Периодически (конфиг, дефолт 5 мин) листит открытые PR и сверяет со state store; PR с текущим `head_sha` без `done`-ревью → повторно в очередь | К-2 |
| СТ-30 | Идемпотентен: не ставит дубль, если событие уже `queued`/`processing` | К-4 |
| СТ-31 | Ловит пропущенные webhook'и, упавших воркеров, восстановление после аутейджа — «пропустили» → «доставили позже» | К-2 |
| СТ-32 | PR без ревью после M циклов (конфиг) → эскалация (коммент + метрика), не бесконечная тихая петля | К-1 |

## §10. Observability и alerting

| СТ | Требование | ← |
|---|---|---|
| СТ-33 | Метрики: `queue_depth, processing_latency{p50,p95}, success_rate, llm_error_rate, llm_429, dead_letter_total, circuit_open, reconcile_requeues` | К-5 |
| СТ-34 | Структурные логи с корреляцией по `delivery_id` от приёма до публикации | К-5 |
| СТ-35 | Алерты: `dead_letter>0`, `success_rate<порог`, `p95>порог`, `circuit_open`, рост очереди без спада | К-1, К-5 |

## §11. Конфигурация

| Ключ | Назначение | Дефолт |
|---|---|---|
| `queue.max_attempts` | попыток до dead-letter | 5 |
| `queue.visibility_timeout` | возврат неподтверждённого сообщения | 2× per-task таймаут |
| `worker.task_timeout` | per-task таймаут анализа | 90 c |
| `state.terminal_deadline` | макс. время вне терминала (СТ-13) | 30 мин |
| `gateway.rate_limit_per_key` | token-bucket | по плану провайдера |
| `gateway.circuit_threshold` | ошибок до размыкания | 5 за 60 c |
| `sweeper.interval` | период reconcile | 5 мин |
| `sweeper.max_cycles` | циклов до эскалации (СТ-32) | 6 |
| `alert.webhook_url` | внешний алерт (issue #2) | пусто = выкл |

## §12. Negative flows / обработка ошибок

| Ситуация | Требуемое поведение | СТ |
|---|---|---|
| Аутейдж провайдера | ретрай+backoff → dead-letter+коммент; reconcile добьёт после восстановления | СТ-15, 21, 27, 31 |
| GitHub API `429` | backoff-ретрай публикации; не терять результат | СТ-20, 26 |
| Падение воркера в `processing` | передоставка по visibility-timeout | СТ-17 |
| Брокер недоступен на приёме | ingress `5xx` → GitHub повторит доставку | СТ-3 |
| Дубль-доставка webhook | dedup по `delivery_id`, no-op | СТ-2, 16 |
| Устаревший `head_sha` (новый пуш) | ревью только для актуального SHA; старое событие → `done`/skip | СТ-11, 16, 29 |
| Слишком большой дифф | клип по токен-лимиту gateway, не падать молча | СТ-19, 24 |
| Prompt-инъекция в теле PR/issue | санитайзинг/изоляция недоверенного ввода в промпте | СТ-19 (Фаза 5) |
| Событие «зависло» вне терминала | по `terminal_deadline` → `failed`+повторная постановка | СТ-13 |

## §13. Верификация (как проверяем)

| Проверка | Критерий | Покрывает |
|---|---|---|
| Chaos: kill воркера в `processing` | событие передоставлено и завершено, дублей нет | СТ-15, 17, 25 |
| Blackhole провайдера на T минут | события ретраятся → dead-letter+коммент; после восстановления reconcile добивает; тихой потери 0 | СТ-15, 27, 31 |
| Дубль-доставка того же `delivery_id` | вторая обработка не запускается | СТ-2, 16 |
| Нагрузочный тест на целевом rps | p95 < 10 мин, success ≥ 99.9% (с ретраями) | СТ-18, 33 |
| Инъекция «зависания» в терминал | по `terminal_deadline` событие покидает не-терминал | СТ-13 |
| Пропуск webhook (отключить ingress на минуту) | reconcile обнаружит PR без ревью и дозапустит | СТ-29, 31 |
| Провал N раз подряд | dead-letter + видимый коммент + метрика + [опц.] алерт | СТ-9, 27, 35 |
