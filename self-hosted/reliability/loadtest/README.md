# Нагрузочный прогон ядра надёжности

Синтетический load-тест, который гоняет **настоящий** код конвейера
`ingress → durable queue → worker (process/supervisor) → sweeper` через большой поток
событий. Замоканы только внешние зависимости — **LLM Z.AI** (`FakeAnalyze`) и
**GitHub API** (`FakeClient`); всё ядро надёжности работает по-настоящему.

Зачем синтетика, а не реальные PR: 100k+ реальных PR упрутся во вторичные rate-limit
GitHub и в стоимость токенов GLM-5. А проверить нужно именно ядро «не молчать» — оно
не зависит от того, настоящий LLM за `analyze` или фейковый. Живой путь GLM-5
подтверждается отдельным smoke (`scripts/smoke.sh` + один реальный PR).

## Что моделируется

Топология как в проде на одном узле Dokploy: общий SQLite на диске (`reliability.db` +
`queue.db`), пул воркеров, пул продюсеров webhook'ов. **Каждый поток открывает свои
соединения к тем же файлам** — точная модель отдельных процессов ingress/worker на
общем volume (в т.ч. несколько реплик воркера).

Инъекция сбоев детерминирована по хешу бизнес-ключа (ретраи стабильны):

| Класс | Доля | Поведение | Что проверяет |
|---|---|---|---|
| `ok` | ~90% | успех сразу | happy-path, throughput |
| `flaky` | 6% | падает N раз, потом успех | retry + backoff (СТ-12), в итоге DONE |
| `bp` | 2% | Backpressure M раз, потом успех | defer без счёта к DLQ (rate-limit) |
| `perm` | 2% | падает всегда | dead-letter + видимый коммент (СТ-27) |

Плюс повторная доставка части webhook'ов (тот же `delivery_id`) — проверка dedup (СТ-2).

## Инварианты (Фаза A)

- **К-1 нет тихих потерь** — каждое принятое событие в терминале (DONE|DEAD_LETTER);
  `non_terminal == 0`.
- **СТ-6/9** — очередь полностью осушается (`depth == 0`).
- **СТ-16 идемпотентность** — не более одного DONE на бизнес-ключ.
- **СТ-27 не молчать** — `dead_letter == dlq_rows == комментов` (каждый провал видим).
- **учёт** — `done + dead == принято`; `принято == 2×webhooks` (dedup сработал).
- **захваты не текут** — таблица `claims` пуста.
- **СТ-7** — обслужены все партиции (репозитории).

Меряются throughput (соб/с на приёме и end-to-end) и латентность p50/p95/p99
(created_at → updated_at из стора).

## Фаза B — reconciliation sweeper (бэкстоп)

Отдельно и детерминированно (управляемые часы): застрявшее событие → ре-queue → DONE;
отравленное (attempts≥max) → DEAD_LETTER + коммент; пропущенный webhook (PR открыт,
события нет) → reconcile → DONE (СТ-13/29/32).

## Запуск

```bash
cd self-hosted
python3 reliability/loadtest/run_loadtest.py --events 100500 --out reliability/loadtest/RESULTS.md
```

Полезные флаги: `--workers`, `--producers`, `--repos`, `--max-depth` (backpressure
продюсеров — держит бэклог ограниченным, как реальный steady-state), `--perm-fail-pct`,
`--flaky-pct`, `--backpressure-pct`, `--keep-db`. Exit code ≠ 0 при любом нарушении
инварианта → пригоден для CI. Результаты последнего прогона — в [`RESULTS.md`](RESULTS.md).

## Что нашёл этот прогон

Первый же прогон вскрыл **реальный баг гонки при горизонтальном масштабировании**:
`DurableQueue.lease` захватывал сообщение как `SELECT` кандидата → затем `UPDATE`.
Внутри одного процесса это защищал `threading.Lock`, но между несколькими
**worker-репликами** (отдельные процессы = отдельные соединения к одному файлу) это
TOCTOU: две реплики читали строку как свободную и обе её лизали → двойная обработка,
гонка переходов состояния (`IllegalTransition`), утёкшие захваты бизнес-ключа.

Фикс — атомарный **условный** `UPDATE ... WHERE leased_until IS NULL OR leased_until<=now`
(SQLite сериализует запись → строку получает ровно один; проигравший перевыбирает
кандидата). Аналогично защищён poison-guard перевод в DLQ. Регресс закреплён тестом
`test_no_message_leased_twice_under_replicas` (две реплики очереди на одном файле,
500 сообщений, 6 потоков — ни одного двойного захвата, ни одной потери).
