# Оценка качества ревью — «посевные баги» (способ A, Q8)

Делает качество ревью **измеримым числом**, чтобы не приходилось субъективно
judge'ить код. Берём набор патчей с ЗАРАНЕЕ известными дефектами, прогоняем через
настоящий адаптер ревью (`chunk_review`) и считаем:

- **Recall** — сколько посаженных багов ревью нашло (нашла X из Y).
- **False-positive** — сколько дефектов ревью выдумало на ЧИСТОМ коде.

Оценщику (PO) judge'ить код не нужно — на выходе проценты.

## Запуск

```bash
cd self-hosted
python3 reliability/eval/run_eval.py --live    # реальный GLM-5 (нужны OPENAI_KEY/BASE/DEEP_MODEL)
python3 reliability/eval/run_eval.py --demo    # фейк-модель — только проверка «трубы», НЕ замер качества
```

Флаги: `--min-recall` / `--max-fp` (пороги для exit-кода → пригодно для CI),
`--show-findings` (печатать сами находки).

## Как это устроено

- `reliability/quality_eval.py` — датасет `DEFAULT_CASES` (SQL-инъекция, проигнорённая
  ошибка, утечка ресурса, захардкоженный секрет, выход за границу, None-разыменование +
  2 чистых кейса) и **чистый скоринг** (recall/FP), покрытый юнит-тестами.
- `run_eval.py` — CLI: `--live` берёт `chunk_review.glm_model_call` (GLM-5), `--demo` —
  канонический фейк.
- Детекция «нашла баг» — по ключевым словам ожидаемой находки (простая, объективная).
  Для более строгой оценки — **модель-судья** (способ B), followup.

## Как пользоваться для калибровки промпта

1. Прогнать `--live` → получить базовый recall/FP на текущем (черновом) промпте.
2. Поправить промпт в `chunk_review.py` (SYSTEM_PROMPT / DEFAULT_REVIEW_INSTRUCTIONS).
3. Перепрогнать `--live` → сравнить: вырос ли recall, не выросли ли ложные.
4. Команда расширяет `DEFAULT_CASES` под свой стек (языки/типы багов).

> Датасет иллюстративный. Чем он ближе к реальным багам вашего кода, тем полезнее
> число. Реальные патчи с историческими багами (из закрытых issue) — лучший источник.
