## 2. As-Is (текущее поведение, с якорями)

- Webhook → `ingress` (HMAC+dedup) → `enrich` (дозаполнение `head_sha`) → durable queue (партиция = repo) → `worker` лизует и делает **один** вызов `analyze(event)` через Gateway → pr-agent (`/describe`, `/review`) → GLM-5 → публикация. `[S3,S4,S7]`
- Таймаут — единый per-task (`RELIABILITY_TASK_TIMEOUT=90`), **без учёта размера диффа**. `[S4,S9]`
- Gateway — один провайдер `zai`, **тиринга моделей нет** (дешёвая/сильная модель не разделены). `[S5]`
- При провале/таймауте — ретрай очередью + видимый dead-letter коммент (СТ-27). Прогресс/ожидание пользователю **не сообщается**. `[S6,S8]`
- На большом диффе GLM-5 не укладывается в 90с → `litellm.Timeout` (наблюдаемо: 271.79с с внутренними ретраями litellm). `[S1]`

## 4. Границы системы (черновик — уточнить на стадии draft)

- **ВНУТРИ:** классификация размера диффа; маршрутизация small/large; быстрая первичная ОС; разбиение большого диффа и синтез единого ревью; тиринг моделей; инкрементальная публикация; кэш по blob-sha.
- **СНАРУЖИ:** сам pr-agent/промпты Qodo (используем как есть); хостинг моделей Z.AI; ёмкость/лимиты Z.AI; GitHub API rate-limit (учитываем, но не управляем).

## 5. Ключевые ограничения и предпосылки

- GLM-5 через Z.AI: латентность растёт с размером промпта; практический предел одного вызова `[УТОЧНИТЬ — замерить порог токенов/строк, при котором P95 < таймаута]`. `[S1]`
- litellm имеет **собственные внутренние ретраи** — при таймауте множит время (90→271с); очередь надёжности ретраит поверх → двойной ретрай. `[S1]`
- Вложенность таймаутов инвертирована: `CONFIG_AI_TIMEOUT(90) > ATTEMPT_TIMEOUT(75)` → gateway бросает поток раньше, чем litellm сам отменится. `[S9]`
- Партиции очереди — по repo; для честного fan-out подзадач одного PR нужна суб-партиция `[УТОЧНИТЬ схему партиционирования]`. `[S7]`
- Тиринг моделей и кэш ревью — **в контракте есть (СТ-24, СТ-19..23), в коде нет**. `[S5,S8]`

## 6. Связанные требования (каскад → эпик)
