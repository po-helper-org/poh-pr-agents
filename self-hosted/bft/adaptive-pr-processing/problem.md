# Проблема — БФТ APRP: Адаптивная обработка PR по размеру диффа

> Стадия 1 pipeline. Здесь **диагноз**: As-Is → Gap, с якорями. **Решения нет** —
> map-reduce/тиринг/fast-ack как способы решения появятся на стадии 2 (concept).
> Якоря — из context-pack (S1–S10).

## 1. As-Is — как система обрабатывает PR сейчас

- Обработка события **одношаговая и не зависит от размера диффа**: воркер лизует
  событие и делает **один** вызов `analyze(event)` → pr-agent (`/describe`,
  `/review`) → GLM-5 → публикация. `[S4]`
- Таймаут — **единый per-task** `RELIABILITY_TASK_TIMEOUT=90`, одинаковый для PR из
  3 строк и из 5000. `[S4,S9]`
- LLM Gateway — **один провайдер `zai`**, без разделения «дешёвая/сильная» модель:
  и триаж, и глубокое ревью, и (гипотетическая) склейка пошли бы одной моделью. `[S5]`
- Пользователю **не сообщается ничего** между приёмом webhook и публикацией
  результата: ни «принято», ни «PR большой, идёт дольше». Видимый коммент
  появляется только при провале (dead-letter, СТ-27) или по готовности ревью. `[S6,S8]`
- Ключ идемпотентности `Event.business_key` включает `head_sha` — на новый пуш
  ревью считается заново **целиком**, ре-использования ревью неизменённых файлов нет. `[S7]`

## 2. Симптомы (наблюдаемая боль)

| # | Симптом | Якорь |
|---|---|---|
| P1 | На большом диффе GLM-5 не укладывается в 90с → `litellm.Timeout` (наблюдаемо 271.79с из-за внутренних ретраев litellm) — **ревью не публикуется** | `[S1]` |
| P2 | До срабатывания таймаута/ретраев автор PR видит **тишину** (или только «Preparing review…»), непонятно, работает ли система | `[S1,S6]` |
| P3 | Даже если большой промпт «влезает», качество ревью на огромном диффе **деградирует** (внимание модели размывается по 5000 строк) `[УТОЧНИТЬ — подтвердить замером качества]` | `[S2]` |
| P4 | Впустую жжётся время/квота Z.AI: 271с на один упавший вызов из-за двойного ретрая (litellm + очередь) | `[S1,S9]` |

## 3. Корневые причины (диагноз, не решение)

| # | Причина | Якорь |
|---|---|---|
| R1 | **Обработка не различает размер PR** — единственный путь рассчитан на «умещается в один вызов» | `[S4]` |
| R2 | **Единый таймаут** не отражает, что стоимость ревью зависит от объёма диффа | `[S4,S9]` |
| R3 | **Нет тиринга моделей** — нечем сделать дешёвую быструю операцию (триаж/оповещение), всё идёт через тяжёлую GLM-5 | `[S5,S8]` |
| R4 | **Контракт «не молчать» покрывает только провал**, но не «ожидание/прогресс» — нет первичной ОС | `[S6,S8]` |
| R5 | **Двойной ретрай**: litellm ретраит внутри + очередь ретраит снаружи; вложенность таймаутов инвертирована (`ai=90 > attempt=75`) | `[S1,S9]` |
| R6 | **Нет ре-использования** результата ревью по неизменённым файлам между пушами (ключ по `head_sha` считает заново весь PR) | `[S7]` |

## 4. Gap-анализ (As-Is → чего не хватает; **без способа реализации**)

| Аспект | As-Is | Gap (чего не хватает) |
|---|---|---|
| Классификация PR | размер не измеряется | нет сигнала «этот PR большой/малый» до обработки |
| Маршрутизация | единственный путь | нет разделения лёгкого и тяжёлого пути обработки |
| Первичная ОС | тишина до результата/провала | нет раннего оповещения автора об ожидании и причине задержки |
| Обработка большого диффа | один вызов на весь дифф | нет способа удержать каждый вызов в пределах таймаута и качества |
| Модели | одна модель на всё | нет дешёвого уровня для лёгких операций |
| Прогресс | нет | автор не видит, сколько уже отревьюено и сколько осталось |
| Повторные пуши | пересчёт всего PR | нет ре-использования ревью неизменённых частей |
| Ретраи/таймауты | двойной ретрай, инверсия вложенности | нет чистой отмены одного вызова до внешнего ретрая |

## 5. Влияние (impact)

- **Автор PR**: на больших PR ревью может не прийти вовсе (P1) и/или прийти как
  провал; UX «система молчит» (P2). `[S1]`
- **Качество**: на грани таймаута ревью либо не публикуется, либо деградирует (P3). `[S2]`
- **Стоимость/ёмкость**: впустую сожжённое время Z.AI (P4), риск упереться в лимиты
  при масштабировании. `[S1]`
- **Доверие к контракту надёжности**: «не молчать» формально соблюдён (будет
  dead-letter коммент), но по духу пользователь долго не понимает, что происходит. `[S6,S8]`

## 6. Что НЕ является этой проблемой (границы диагноза)

- Качество промптов pr-agent/Qodo как таковых — вне scope (используем как есть). `[S2]`
- Латентность/лимиты самого Z.AI — внешние, не управляем (учитываем как ограничение).
- Малые PR — текущий одиночный проход их обрабатывает корректно, регресс не нужен. `[S4]`
