# Концепты решения — БФТ APRP

> Стадия 2 pipeline. Здесь **формы решения** (не требования — те на стадии 4).
> Каждый концепт закрывает Gap из `problem.md §4`. Предварительный вердикт → в дебаты
> (стадия 3, Адвокат Дьявола) до фиксации требований.

## Общая рамка (единый шаг для всех концептов)

**Классификатор размера** на шаге `enrich` (там уже дозаполняется `head_sha` до
очереди) считает `diff-weight` — файлы / изменённые строки / оценка токенов —
дёшево, без LLM, и присваивает **класс размера** событию. Дальше концепты
расходятся в том, что делать с `large`. `[S3,S7]`

---

## Концепт A — Size-routing + fast-ack + map-reduce + model-tiering (на слое надёжности)

**Идея.** `small` → текущий одиночный проход (fast-path, регресса нет). `large` →
heavy-path:
1. **fast-ack**: сразу публикуем шаблонный/дешёвый коммент «PR большой (N файлов,
   ~M строк), ревью по частям, ETA ~T» через идемпотентный upsert (СТ-25). `[S6]`
2. **planner → fan-out**: разбиваем дифф на **чанки** (по файлу/логическому блоку,
   каждый ≤ порога токенов), кладём как отдельные подзадачи в durable queue
   (суб-партиция на PR для честности), заводим запись job(PR, N чанков). `[S7]`
3. **deep review чанков**: каждый чанк ревьюит GLM-5 (deep-tier) **в пределах
   таймаута** — маленький фокусный промпт → выше качество, нет таймаута. Параллелизм
   ограничен rate-limit Gateway. `[S5,S8]`
4. **fan-in → reduce**: когда все чанки DONE (барьер по job) — reduce-задача
   синтезирует единое ревью (дедуп, группировка, приоритет), дешёвой моделью или
   GLM-5 по компактным findings (не по сырому диффу). `[S8]`
5. **инкрементальная публикация**: коммент прогресса обновляется по мере готовности
   чанков («отревьюено 3/12») через тот же upsert. `[S6]`
6. **тиринг моделей** (СТ-24, СТ-19..23): cheap-tier — триаж/ack/склейка; deep-tier
   (GLM-5) — глубокое ревью приоритетных чанков. `[S5,S8]`
7. (опц., Q7) **кэш по blob-sha**: на повторном пуше ре-ревьюятся только изменённые
   файлы. `[S7]`

**Плюсы:** каждый вызов ограничен → нет таймаута и деградации; переиспользует примитивы
слоя надёжности (очередь, fan-out/fan-in, upsert, gateway-тиринг); «не молчать»
расширен на ожидание/прогресс; частичное ревью возможно (см. Q5); масштабируется.
**Минусы/риски:** новый примитив координации fan-in (таблица `jobs` + барьер);
кросс-файловые находки требуют качественного reduce; больше движущихся частей;
стоимость (много вызовов) — нужен бюджет/приоритизация чанков (Q6).

---

## Концепт B — «Только таймауты + fast-ack», без разбиения (минимальный)

**Идея.** Не делить дифф. Поднять/упорядочить таймауты (`CONFIG_AI_TIMEOUT` →
внутренний и самый короткий, `attempt`/`task`/`visibility` — выше по вложенности),
**отключить внутренние ретраи litellm** (очередь и так ретраит), добавить fast-ack. `[S1,S9]`

**Плюсы:** минимальная правка, чинит немедленную боль на средних PR; убирает двойной
ретрай (271с). **Минусы:** **не решает по-настоящему большие диффы** — упрёмся в
потолок контекста/качества GLM-5 (P3); длинная занятость воркера одним PR; не
масштабируется. Годится как **немедленная митигация**, не как целевое решение.

---

## Концепт C — Иерархическая сводка → ревью по сводкам (summarize-then-review)

**Идея.** Дешёвая модель делает сводку каждого файла → GLM-5 ревьюит по сводкам +
точечным диффам, не видя весь сырой дифф.

**Плюсы:** вход GLM-5 всегда мал. **Минусы:** сводка **теряет детали** → пропуск
строковых багов; для код-ревью (где важна конкретная строка) риск качества высок.
Хорошо как **техника внутри reduce** (сжать findings), но не как основной путь ревью.

---

## Сравнение

| Критерий | A (routing+map-reduce+tiering) | B (только таймауты) | C (summarize-then-review) |
|---|---|---|---|
| Решает большие диффы | ✅ да | ⚠️ частично | ✅ вход мал |
| Качество ревью | ✅ высокое (фокус) | ⚠️ деградирует на огромных | ❌ риск пропусков |
| Первичная ОС (fast-ack) | ✅ | ✅ | ✅ |
| Сложность внедрения | ⚠️ высокая (fan-in) | ✅ низкая | ⚠️ средняя |
| Стоимость (вызовы) | ⚠️ выше (нужен бюджет) | ✅ низкая | ✅ низкая |
| Масштабируемость | ✅ | ❌ | ⚠️ |
| Переиспользует слой надёжности | ✅ сильно | ✅ | ⚠️ частично |

## Предварительный вердикт (в дебаты)

**Основной — A**, с **B как немедленной митигацией** (правка таймаутов + отключение
litellm-ретраев внедряется сразу и живёт под A) и **C как техникой внутри reduce**
(сжатие findings, не замена ревью). Открытые параметры для дебатов/уточнения:
пороги (Q2), доступность дешёвой модели (Q3), SLA (Q4), атомарность vs частичное
ревью (Q5), приоритизация/бюджет чанков (Q6), кэш в v1 или followup (Q7).

**Красные зоны для стадии 3 (Адвокат Дьявола):** консистентность единого ревью из
независимых чанков; поведение при частичном отказе чанков (не молчать per-chunk);
стоимость/лимиты Z.AI при fan-out; идемпотентность инкрементального коммента при
ретраях/reconcile; честность суб-партиций одного большого PR против остальных.

## Adversarial Review (стадия 3 — 3 раунда, субъект: Концепт A)

### Раунд 1 — Защита

**Аналитик (почему A корректен и достаточен):**
1. Ограниченный размер каждого вызова снимает первопричину P1/P3 — таймаут и
   деградацию (маленький фокусный промпт). `[S1, problem R1/R2]`
2. Не изобретает инфраструктуру: fan-out/fan-in на durable queue, единый коммент
   через upsert, тиринг на Gateway — всё уже есть/запланировано. `[S5,S6,S7,S8]`
3. «Не молчать» усиливается: fast-ack закрывает P2 с нулевой секунды. `[S6, СТ-27]`
4. Независимые чанки-подзадачи сохраняют К-1: каждый чанк ретраится/уходит в DLQ
   отдельно, без тихой потери. `[S7,S8]`

**Адвокат Дьявола (атака по линзам):**
- **Нагрузка:** fan-out PR на 200 файлов = 200 подзадач в одну суб-очередь → может
  **голодить** PR других репозиториев; честность суб-партиций одного PR не задана. `[S7]`
- **Отказ Z.AI (mid-job):** часть чанков DONE, часть падает; circuit breaker
  размыкается → **reduce может не собраться никогда**, job зависает. `[S5]`
- **Идемпотентность/гонки:** инкрементальный коммент правят несколько чанк-воркеров
  параллельно → **гонка за коммент** (потерянные апдейты/дубли); fan-in reduce при
  передоставке может выполниться **дважды**. `[S6,S7]`
- **Изменение требований:** через 6–12 мес добавят провайдера/модель — тиринг не
  должен быть захардкожен.
- **ПДн/Compliance:** чанкинг шлёт код во внешний LLM (Z.AI); fan-out **умножает**
  число внешних вызовов/эгресс кода. Данные-резиденция? `[УТОЧНИТЬ у PO/безопасности]`
- **Скрытые зависимости:** качество reduce зависит от формата findings; **кросс-файловые
  баги** невидимы почанково.
- **Трассировка/стоимость:** без бюджета fan-out стоимость не ограничена (Q6).

### Раунд 2 — Пересмотр

**Аналитик (ответы + усиления → правки в требования стадии 4):**
- **M1 (нагрузка):** суб-партиция на PR **+ потолок одновременных чанков на PR** →
  один большой PR не голодит остальные (новое НФТ честности).
- **M2 (отказ Z.AI):** **partial-reduce** — reduce собирается по чанкам, готовым к
  дедлайну; непройденные файлы **явно перечислены** в ревью («не удалось отревьюить
  X — повтор автоматически/вручную»). «Не молчать» **по-чанково**. circuit-open →
  fast-ack уже предупредил, dead-letter эскалация остаётся. `[СТ-27]`
- **M3 (гонка за коммент):** чанк-воркеры пишут findings **в стор job**, а не в
  коммент; коммент правит **единственный писатель** (reduce/coordinator) идемпотентным
  upsert → гонки нет. `[S6]`
- **M4 (двойной reduce):** fan-in **идемпотентен** через CAS по состоянию job (ровно
  один reduce на job); передоставка → skip. `[S7]`
- **M5 (тиринг):** конфиг провайдеров/моделей **data-driven** (список Provider в
  Gateway), не хардкод. `[S5,S8]`
- **M6 (бюджет):** **потолок чанков/токенов на PR + приоритизация** (core > tests >
  generated, исключение vendored/generated) + **overflow-политика** (ревьюим приоритетное
  в пределах бюджета, остаток — явной пометкой). `[Q6]`

**Адвокат Дьявола (поверка усиленного):**
- Принимает M1–M6 **при условии, что они станут явными требованиями** (ФТ/НФТ), а не
  ремаркой. Новый вопрос: «PR больше max-чанков даже после приоритизации?» → overflow
  M6 обязана это покрыть (ревью топ-приоритета + пометка). ПДн-резиденция остаётся
  `[УТОЧНИТЬ]`, не блокирует концепт, но обязана попасть в «Открытые вопросы» draft.

### Раунд 3 — Вердикт

**Аналитик:** A закрывает все Gap и, с M1–M6, устойчив к нагрузке, отказу Z.AI и
гонкам; B входит как немедленная митигация, C — техника внутри reduce.
**Адвокат Дьявола:** возражения адресованы переводом в явные требования; неприемлемых
немитигированных рисков не осталось; ПДн-резиденция — открытый вопрос, не блокер.

**Вердикт модератора: ПРИНЯТ С МОДИФИКАЦИЯМИ.** Концепт A идёт в draft (стадия 4) со
следующими обязательными к кодификации модификациями:

| M | Модификация | Тип требования (draft) |

| M6 | Бюджет чанков + приоритизация + overflow-политика | ФТ + НФТ |
